SOAP = Simple Object Access Protocol
Protocole de messagerie basé sur XML, standardisé par le W3C (2003). Permet l'échange de messages structurés entre applications dans un environnement distribué — indépendant du langage, de la plateforme et du système d'exploitation.
🏭
Qui l'utilise ? Banques mondiales, hôpitaux, gouvernements, opérateurs télécom, ERP SAP/Oracle. Partout où la sécurité, la fiabilité et les contrats stricts sont non-négociables.
📅
Historique : XML-RPC (1998, Microsoft) → SOAP 1.0 (2000, W3C Note) → SOAP 1.2 (2003, W3C Recommandation officielle). Supporté par Microsoft .NET, IBM WebSphere, SAP, Oracle, Apache Axis2…
⚙️ Pourquoi SOAP plutôt qu'une solution maison ?
1
Standard universel
Tous les langages (Java, .NET, Python, COBOL) produisent et consomment SOAP identiquement grâce au standard W3C.
2
Sécurité niveau message
WS-Security chiffre le contenu indépendamment du transport — même sans HTTPS, le message reste protégé.
3
Transactions ACID
WS-AtomicTransaction garantit la cohérence sur plusieurs systèmes : tout réussit ou tout est annulé.
4
Contrat WSDL auto-documenté
Le WSDL décrit formellement le service — les outils génèrent le code client automatiquement.

<soap:Envelope> OBLIGATOIRE
Élément racine — identifie le document XML comme message SOAP. Définit les espaces de noms.
<soap:Header> OPTIONNEL
Métadonnées du message : authentification, routage, transaction. 1er enfant de l'Envelope.
<soap:Body> OBLIGATOIRE
Contenu principal du message — requête ou réponse de l'opération métier.
<soap:Fault> DANS LE BODY
Signale une erreur de traitement. Code + Reason + Detail.

SOAP est structuré comme une enveloppe postale :
• L'Envelope = l'enveloppe
• Le Header = les informations d'acheminement
• Le Body = le contenu de la lettre
• Le Fault = le retour en cas d'erreur de livraison
📄 message_soap_complet.xml — Requête bancaire avec sécurité 
<?xml version="1.0" encoding="UTF-8"?>
<soap:Envelope
  xmlns:soap="http://www.w3.org/2003/05/soap-envelope"
  soap:encodingStyle="http://www.w3.org/2003/05/soap-encoding">

  <!-- ①  HEADER : métadonnées (optionnel) -->
  <soap:Header>
    <wsse:Security xmlns:wsse="http://docs.oasis-open.org/wss/...">
      <wsse:UsernameToken>
        <wsse:Username>koffi.ama@banque.tg</wsse:Username>
        <wsse:Password>••••••••</wsse:Password>
      </wsse:UsernameToken>
    </wsse:Security>
  </soap:Header>

  <!-- ②  BODY : contenu principal (obligatoire) -->
  <soap:Body>
    <bank:ConsulterSolde xmlns:bank="http://banque.tg/services/v2">
      <bank:NumeroCompte>TG53TG8199802100040000000000</bank:NumeroCompte>
      <bank:Devise>XOF</bank:Devise>
    </bank:ConsulterSolde>
  </soap:Body>

</soap:Envelope>

WSDL = Web Services Description Language — Le contrat du service. Décrit QUOI fait le service (opérations, types de données, adresse réseau). SOAP décrit COMMENT les messages sont échangés. Ensemble = système complet et auto-documenté.
🏗 Les 5 blocs du WSDL
<types>
Types de données (XSD)
Définit les structures XML attendues dans les messages — utilise XSD.
<msg>
Messages
Décrit les paramètres d'entrée et de sortie de chaque opération.
<port>
PortType — Interface
Liste les opérations disponibles = l'interface abstraite du service.
<bind>
Binding
Lie les opérations au protocole SOAP et au transport HTTP.
<svc>
Service — Adresse réseau
Donne l'URL endpoint du service pour les appels réels.
🤖
Génération automatique de code !
Avec le WSDL, des outils comme wsimport (Java), wsdl.exe (.NET) ou SoapUI génèrent automatiquement tout le code client — on n'écrit jamais le XML SOAP à la main en production.
📜 service_banque.wsdl — Extrait complet 
<wsdl:definitions
  xmlns:wsdl="http://schemas.xmlsoap.org/wsdl/"
  xmlns:soap="http://schemas.xmlsoap.org/wsdl/soap/"
  xmlns:xs="http://www.w3.org/2001/XMLSchema"
  targetNamespace="http://banque.tg/services">

  <!-- 1. Types XSD des données -->
  <wsdl:types>
    <xs:schema targetNamespace="http://banque.tg/services">
      <xs:element name="ConsulterSolde">
        <xs:complexType><xs:sequence>
          <xs:element name="NumeroCompte" type="xs:string"/>
          <xs:element name="Devise" type="xs:string"/>
        </xs:sequence></xs:complexType>
      </xs:element>
    </xs:schema>
  </wsdl:types>

  <!-- 2. Messages (requête + réponse) -->
  <wsdl:message name="SoldeRequest">
    <wsdl:part name="body" element="tns:ConsulterSolde"/>
  </wsdl:message>

  <!-- 3. Interface (opérations disponibles) -->
  <wsdl:portType name="BanqueService">
    <wsdl:operation name="ConsulterSolde">
      <wsdl:input  message="tns:SoldeRequest"/>
      <wsdl:output message="tns:SoldeResponse"/>
    </wsdl:operation>
  </wsdl:portType>

  <!-- 4. Binding SOAP -->
  <wsdl:binding name="BanqueSOAPBinding" type="tns:BanqueService">
    <soap:binding style="document"
      transport="http://schemas.xmlsoap.org/soap/http"/>
  </wsdl:binding>

  <!-- 5. Adresse du service -->
  <wsdl:service name="BanqueService">
    <wsdl:port name="BanqueSOAP" binding="tns:BanqueSOAPBinding">
      <soap:address location="https://api.banque.tg/soap/v2"/>
    </wsdl:port>
  </wsdl:service>

</wsdl:definitions>
🔒
Les extensions WS-* — Ce qui fait de SOAP le standard incontournable de l'entreprise. Ces extensions s'intègrent dans les en-têtes SOAP sans modifier la structure de base du protocole.
🔒 WS-Security — Sécurité niveau MESSAGE
Chiffrement XML Encryption + signature XML Signature au niveau du message — indépendant du transport. Jetons : Username, X.509, SAML. Requis par HIPAA, PCI-DSS.
ws-security-header.xml
<soap:Header>
  <wsse:Security xmlns:wsse="...wss-secext-1.0.xsd">
    <xenc:EncryptedKey Id="EK-001">
      <xenc:CipherData>
        <xenc:CipherValue>ABC123XYZ...</xenc:CipherValue>
      </xenc:CipherData>
    </xenc:EncryptedKey>
    <ds:Signature xmlns:ds="...xmldsig#">
      <!-- Signature numérique du message -->
    </ds:Signature>
  </wsse:Security>
</soap:Header>
✅ Même sans HTTPS, le contenu reste chiffré et infalsifiable
📨 WS-ReliableMessaging — Livraison garantie
Accusés de réception, réémission automatique, déduplication. Un virement bancaire ne peut pas se perdre.
→ Client envoie message + numéro de séquence
→ Serveur accuse réception (AcknowledgementRange)
⚠ Si pas d'accusé → réémission automatique
✅ Déduplication : doublons détectés et ignorés
⚛️ WS-AtomicTransaction — Transactions ACID distribuées
Transactions ACID sur plusieurs systèmes distants. Si un système échoue → rollback global de tous les autres. REST n'a pas d'équivalent natif.
Exemple : Virement entre 2 banques
ÉTAPE 1Débiter compte source (Banque A)
ÉTAPE 2Créditer compte dest. (Banque B)
ERREURBanque B indisponible → panne réseau
ROLLBACKDébit Banque A annulé automatiquement ✅
🗺️ WS-Addressing — Routage des messages
Standardise les adresses source/destination dans le Header. Permet le routage via intermédiaires et les communications asynchrones dans les architectures SOA.
ws-addressing-header.xml
<wsa:To xmlns:wsa="...2005/08/addressing">
  https://api.banque.tg/soap/virement
</wsa:To>
<wsa:ReplyTo>
  <wsa:Address>https://client.app.tg/callback</wsa:Address>
</wsa:ReplyTo>
<wsa:MessageID>uuid:550e8400-e29b-41d4</wsa:MessageID>

🏦 Quand choisir SOAP ?
✓
Sécurité de niveau entreprise requise
Chiffrement message-level, conformité PCI-DSS, HIPAA, réglements bancaires.
✓
Transactions distribuées critiques
Virements, réservations multi-systèmes où un échec partiel est inacceptable.
✓
Intégration de systèmes legacy
ERP SAP/Oracle, systèmes mainframe, plateformes télécom OSS/BSS.
🚀 Quand choisir REST ?
✓
APIs publiques grand public
Applications mobiles, startups, plateformes web modernes, microservices.
✓
Besoin de rapidité de développement
JSON natif JS/Python, pas de génération de stubs, tests avec curl/Postman.
✓
Scalabilité horizontale massive
Stateless = facile à scaler. IoT, streaming, applications haute charge.
🏦 Banque & Finance
SWIFT ISO 20022 — Format MX basé sur SOAP. Virements interbancaires mondiaux. Adoption universelle en cours (2023-2025).
PCI-DSS — Norme carte bancaire. WS-Security obligatoire pour le chiffrement des données de paiement.
Cas concret — Virement Ecobank → ORABANK via SOAP : token chiffré WS-Security + WS-AtomicTransaction garantit la cohérence.
🏥 Santé Numérique
HL7 v3 — Standard mondial d'échange de données médicales. Messages structurés SOAP entre hôpitaux.
HIPAA — Loi US sur la confidentialité médicale. WS-Security obligatoire pour les échanges de données patient.
Stateful — Une consultation multi-étapes (auth médecin → dossier → prescription) maintenue en session SOAP.
🏛 Gouvernement
DGI Togo/Bénin — Déclarations fiscales électroniques via services SOAP. WSDL garantit format exact des données.
État civil numérique — Échanges inter-registres (naissance, mariage, décès) sécurisés par WS-Security.
e-Gov — Portails gouvernementaux interconnectés via bus de services SOA/SOAP dans de nombreux pays.
📡 Télécommunications
OSS/BSS — Systèmes d'opération et de support business. Orange, MTN, Togocom : historiquement SOAP.
Mobile Money — Transferts inter-opérateurs (Flooz ↔ T-Money) via services SOAP pour la cohérence transactionnelle.
Recharge — APIs de recharge à la demande, activation de services via SOAP/WSDL entre MVNO et opérateurs.
⚙️ ERP & Entreprise
SAP PI/PO — Process Integration / Orchestration. Connecte SAP aux systèmes tiers via SOAP. Standard dans les grandes entreprises.
Oracle SOA Suite — Orchestre les processus métier cross-systèmes via des services SOAP dans les architectures SOA.
Microsoft Dynamics — Expose ses fonctions CRM/ERP via SOAP/WSDL pour l'intégration avec d'autres systèmes.
💡 Pourquoi SOAP persiste ?
Coût de migration — Remplacer un système SOAP bancaire en production = risque et coût énormes. Les systèmes durent 20-30 ans.
Conformité réglementaire — PCI-DSS, HIPAA, GDPR imposent des niveaux de sécurité que WS-Security satisfait nativement.
Gagan Mehta (2025) — "SOAP est irremplaçable dans des secteurs comme la banque, les télécoms et les services gouvernementaux."
🌍
En Afrique de l'Ouest — Les systèmes d'interopérabilité bancaire (BCEAO, GIM-UEMOA), les plateformes Mobile Money inter-opérateurs et les systèmes d'information gouvernementaux utilisent massivement SOAP pour ses garanties de sécurité et de fiabilité transactionnelle.
